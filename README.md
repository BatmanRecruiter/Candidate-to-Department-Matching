# Candidate Department Matching

Recruiter-operator tool that scores LinkedIn-style candidate CSVs against the
phData role library and produces a download-ready, template-aligned CSV.

## Stack

Express + Vite + React + Tailwind + shadcn/ui (fullstack-template baseline).
Pure deterministic matcher — no paid APIs.

## What it does

- Upload a CSV of LinkedIn-style candidate profile fields.
- Match each candidate against an embedded phData role library.
- Evaluate career context before matching, so tool mentions alone do not trigger
  a department or role fit.
- Apply strict role gates for required skills, location, years of experience,
  and seniority/management level.
- Route candidates to the best-fit department even when the system cannot
  confidently assign a specific role.
- Export a template-shaped CSV with four appended evaluation fields.
- Optionally save uploads and exports for later re-download using a private
  history key.
- Calibrate recruiter feedback by marking matches correct or entering the right
  department, role, and reasoning. Future uploads with the same candidate key
  apply exact-match calibration overrides.

## Quick start

```bash
npm install
npm run dev           # dev server, port 5000
# or
npm run build
NODE_ENV=production node dist/index.cjs
```

Health check: `GET /api/health` → `{ ok: true, jobs: 42 }`.
Role library JSON: `GET /api/role-library`.

Saved-file endpoints:

- `GET /api/saved-files` requires an `x-history-key` header.
- `GET /api/saved-files/:id` requires the same `x-history-key`.
- `POST /api/saved-files` stores upload/export CSV text using a hashed history
  key. Raw history keys are not persisted.

Calibration endpoints:

- `GET /api/calibrations` requires an `x-history-key` header.
- `POST /api/calibrations` stores recruiter feedback and exact-match overrides
  under the hashed history key.

## Matching contract

- Departments: only the 8 JD departments, `Unsure / Not Enough Information`,
  or `Not a Match for COMPANY`
- Role fit: a specific JD title only when required skills, location, YOE, and
  seniority gates are satisfied; otherwise an Unsure/No-match label
- Skills are supporting evidence only. A candidate must show relevant recent
  work context, title alignment, and actual data/AI/cloud/consulting/product or
  enterprise-sales delivery before being routed.
- Incidental tool usage, unrelated professions, school-affiliated roles, and
  self-owned/company contexts are blocked unless the profile also shows clear
  enterprise data-platform delivery evidence.
- Confidence: `1` (educated guess), `2` (confident dept), `3` (confident dept AND role)
- Output: template columns in order with legacy `AI Score` and `AI Reasoning`
  removed, plus four appended columns:
  `COMPANY Match Confidence`, `COMPANY Fit Rationale`, `COMPANY Department Fit`,
  `COMPANY Role Fit`
- Input row order is preserved

## Security notes

- Saved-file history is protected by a user-provided private history key.
- The database stores only a SHA-256 hash of the history key, not the raw key.
- Saved CSV payloads are capped by the API schema and body parser.
- Calibration feedback is protected by the same private history key approach.
- CSV export cells are sanitized against spreadsheet formula injection.
- `data.db` and related SQLite files are ignored and should not be committed.
